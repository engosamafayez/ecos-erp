## 1. Versioning Philosophy

Database schema versioning in ECOS follows three principles:

1. **Migrations are immutable once merged** — a merged migration is history; never edit it
2. **Schema state is derivable** — running all migrations from scratch on a clean database must produce the correct current schema
3. **Milestones are documented** — major schema states are tagged so the team can communicate clearly about "what version are we at?"

---

## 2. Migration File Naming

Laravel migration naming convention:

```
{timestamp}_{descriptive_name}.php
```

The timestamp is `YYYY_MM_DD_HHMMSS` — generated by Artisan. The descriptive name uses snake_case and must convey the intent:

| Good | Bad |
|---|---|
| `create_products_table` | `products` |
| `add_pending_review_status_to_products` | `update_products` |
| `add_idx_orders_company_id_status` | `add_index` |
| `backfill_products_category_scope` | `fix_products` |
| `rename_categories_type_to_category_scope` | `alter_categories` |

**Rules:**
- Every name must be unique (timestamp guarantees this)
- Name must match the single concern in the migration
- Never prefix with a number (timestamp handles order)

## 3. Schema Version Table

ECOS maintains a `schema_versions` table alongside Laravel's standard `migrations` table. This table records milestone versions, not every migration:

```
Table: schema_versions

id           SERIAL PRIMARY KEY
version      VARCHAR(20) NOT NULL         -- e.g., "1.0.0", "1.1.0"
label        VARCHAR(100) NOT NULL        -- e.g., "Inventory Module Baseline"
migrated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
notes        TEXT NULL
```

### Milestone Version Format

```
{major}.{minor}.{patch}

major  —  Breaking architectural change or full module addition
minor  —  New feature additions; backwards-compatible
patch  —  Bug fix migrations, index additions, constraint additions
```

## 5. Recording a New Milestone

When a significant set of migrations is complete, record a milestone:

```sql
INSERT INTO schema_versions (version, label, notes)
VALUES (
  '1.2.0',
  'Category Scope Separation',
  'Renamed categories.type to category_scope; added cross-scope validation; 3 migrations'
);
```

This is done by the Engineering Lead after verifying all milestone migrations are merged and tested.

## 6. CI Schema Verification

The CI pipeline must verify:

1. **Migration is reversible** — run `php artisan migrate` then `php artisan migrate:rollback`; no errors
2. **Fresh migration matches current** — `php artisan migrate:fresh` followed by `php artisan test` must pass
3. **No missing migration** — developer did not edit schema directly on dev DB without a migration

```
CI Job: schema-verify
  - docker compose up -d db
  - php artisan migrate:fresh
  - php artisan migrate:rollback --step=3
  - php artisan migrate
  - php artisan test --filter=DatabaseIntegrity
```

---

## 7. Version Communication

When discussing schema state across the team, use:

```
"We are on schema 1.1.0"
"This change targets schema 1.2.0"
"This bug exists in all versions before 1.0.0"
```

Never say "the current schema" without a version number in architectural discussions.

---

## 8. Immutable Migration Rule

Once a migration is merged to the main branch:

- **Never edit** the migration file
- If the migration had an error, create a **new corrective migration**
- If a column was added incorrectly, create a new migration to fix it
- Use `VER-001` to `VER-005` governance rules below

---

## 9. Governance Rules

| Rule | Statement |
|---|---|
| **VER-001** | Merged migrations are immutable. A new corrective migration must be created for any fix. |
| **VER-002** | Each migration file addresses one concern only. |
| **VER-003** | Migration names must describe the change, not the table (e.g., `add_cost_price_to_products`, not `update_products`). |
| **VER-004** | Schema milestones are recorded in `schema_versions` after Engineering Lead review. |
| **VER-005** | CI must verify migration freshness and reversibility on every PR that touches a migration file. |
